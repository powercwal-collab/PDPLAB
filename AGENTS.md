# Prototype Instructions

Version history, restore instructions, current Figma node references, and ready-to-use continuation prompts are documented in `agent.md`. Read both files before restoring or extending the project.

Run the local server yourself and open the preview in the browser available to this environment. Do not give the user server-start instructions when you can run it.

Before making substantial visual changes, use the Product Design plugin's `get-context` skill when the visual source is unclear or no longer matches the current goal. When the user gives durable prototype-specific design feedback, preferences, or decisions, record them in `AGENTS.md`.

When implementing from a selected generated mock, treat that image as the source of truth for layout, component anatomy, density, spacing, color, typography, visible content, and hierarchy.

## Durable design direction

- Match the supplied GEOBase dashboard references: white rounded app canvas, pale blue-gray surround, grouped left navigation, pale-blue active rows with electric-blue right rule, thin gray panel borders, generous radii, restrained enterprise analytics.
- Product-specific rule: overall PDP uses stars; individual modules use only 弱 / 较弱 / 中 / 强 / 极强 with coefficients 0 / 0.25 / 0.5 / 0.75 / 1.
- Keep the first screen conclusion-led and focused on score, gaps, module maturity, and the P0 route.
- Overall diagnosis card must keep a spacious left/right composition: numeric score on the left; overall stars, page type, explanation, and T1 target on the right.
- The pre-upload starting state should contain only project selection and PDP content upload as its core tasks.
- The two pre-upload task cards must share one centered 330px content axis: step label, title, and primary control use matching left/right edges and vertical baselines. Project selection and file drop controls use the same 64px height and matching 12px bold primary text.
- The upload project picker and the workbench project switcher must reuse the same menu panel logic and visual states: complete backend project list, selected checkmark, fixed create-project action, internal scrolling, outside-click dismissal, and Escape dismissal.
- Every lightweight dropdown or popover must use the shared dismiss contract: keep trigger and panel in one scoped ref, close on outside `pointerdown`, close on `Escape`, and keep `aria-expanded` synchronized. Reuse this rule for future account, project, filter, language, notification, and similar menus; native selects and blocking modal backdrops retain their own semantics.
- The default product entry is a spacious home screen inspired by the supplied Lovart layout: one prominent PDP upload action plus recent project cards. Only after choosing a project or completing an import should the user enter 项目总览.
- Home is structurally independent from the project workbench and must not show the workbench sidebar. Once inside a project, the top bar must provide an explicit 返回首页 action.
- The project workbench top bar must also provide a fast project switcher so users can jump among recent projects without returning home.
- After a PDP upload completes, always show an independent diagnosis progress screen before score review; do not jump directly into the project overview.
- “导入新版本” from a project workbench must carry and preselect the active project automatically, while keeping the shared project picker available for correction. Switching to another project or creating one requires an explicit ownership-change confirmation; the final upload uses the user's confirmed project. Homepage/new-project upload entry points keep the normal unselected picker.
- Keep a lightweight circular user-management entry in the top-right of the standalone homepage because the workbench sidebar is hidden there; it must reuse the same account menu as the sidebar user card. Position the entry relative to the standalone home app shell, never the browser viewport, so it remains inside the white rounded frame on wide screens; keep its account popover right-aligned to the same entry and contained within the viewport on tablet and phone.
- Current delivery priority is the evaluation loop: module review, score adjustment, completeness validation, locked score versions, history, and overview synchronization must remain functional.
- At 900px and below, score review must be a true single-column flow: module checklist first, evidence/detail card second. Both cards and every title/content block share the same centered page gutter; never allow the later-loaded workflow stylesheet to restore a compressed two-column grid.
- AI diagnosis versions are auto-confirmed and locked after all 11 modules and their evidence pass server validation; do not insert a mandatory human confirmation step. Manual review remains an optional correction flow that creates a new version.
- Keep model output and trusted scoring separate: model adapters return typed evidence, information completeness, T2/T1/T0 visual tier, integration, judgment, confidence, and one discrete coefficient from `0 / 0.25 / 0.5 / 0.75 / 1`; the backend applies deterministic ceilings and calculates module scores, total score, and overall stars.
- Treat `/api/diagnosis-config/` as the frontend source of truth for PDP module names, weights, maturity definitions, rule revision, and star bands. Never introduce a second hardcoded scoring scale in dashboard or review flows.
- Runtime AI model settings and PDP Skill settings must remain separate Django admin models and separate menu entries. Keep API keys and Skill tokens server-side and encrypted; the frontend may show only configured/unconfigured state, active model, rule version, source revision, and safe endpoint metadata.
- Never reuse or scrape local Codex/ChatGPT login sessions, cookies, tokens, or credential files as an OpenAI API key. Production model calls require an independently configured OpenAI Platform credential; Codex-session reviews must be labeled `codex_verified`, not `ai_auto`.
- Project cards must prefer backend `score_label` and `cover_url`; the selected project overview and score history must resolve to the same latest diagnosis version.
- Persisted project cards must never borrow static example artwork or example scores. When no backend cover exists, show the neutral no-source state; normalize legacy ratings through the active backend star bands before serialization; card dates use the newest project, diagnosis, or cover-source activity.
- Dashboard gap ranking, “为什么还不是” copy, P0 cards, and the full optimization route must derive from the selected project's latest module assessments. Use `max - score`, omit zero gaps, sort by gain then maturity/rule order, and bind the selected module's judgment/evidence; never restore hardcoded gap or task examples.
- The dashboard maturity card must expose all 11 modules without increasing its accepted six-row density. Use two manual pages (6 + 5 rows) with clickable pagination dots below the table, retain a fixed table/card height, and reset to page 1 when the active diagnosis data changes.
- The dashboard maturity carousel also supports horizontal mouse dragging and touch swiping, advances every five seconds, and restarts its timer after manual dragging or pagination-dot selection. Preserve vertical page scrolling on touch devices and honor reduced-motion preferences.
- The workbench project switcher must render the complete backend project list, fit up to 20 compact rows before internal scrolling, keep the create-project action outside the scroll region, and close on outside click or Escape.
- The score-diagnosis detail panel must never reuse static example copy across modules. It must bind the selected module's saved `judgment`, page evidence reason/OCR, maturity logic, and current scoring rule `strong_standard`; strong modules show maintenance/validation guidance while weak and medium modules show improvement guidance.
- Score-diagnosis module importance is derived from the active rule weight: modules with a maximum score of 10 or more are labeled `重点模块`, while modules below 10 are labeled `次要模块`. Dashboard score columns must keep headers, numerators, separators, and denominators vertically aligned with tabular numerals.
- Project management edits project name and category together; blank edit fields preserve the current value. Management mode supports multi-select batch deletion with one irreversible confirmation and one owner-scoped backend operation.
- The score-history version list must never grow taller than the adjacent module-detail card. Keep its accepted record density fixed, use internal vertical scrolling for longer histories, and provide a separate delete icon on every version with an irreversible-action confirmation modal before calling the backend delete endpoint.
- The selected score-history version is the active workbench diagnosis context. Switching versions must update project overview, score diagnosis, gap ranking, and optimization tasks; deleting a selected version falls forward to the first remaining record, while deleting another version preserves the current selection.
- The last selected score-history version must remain selected when the user leaves the history page for another workbench view and returns. Reset it only when changing projects or when the selected version is deleted.
- The score-history module detail keeps its summary and column header fixed while module rows scroll internally. Header and row grids must share the same columns, including a per-module evidence thumbnail sourced from saved evidence; clicking a thumbnail opens a modal preview, while versions without saved evidence show a truthful no-slice state.
- Evidence thumbnails may crop for scanability, but the score-history evidence modal must preserve the complete evidence image at its natural aspect ratio. Overflow belongs inside the modal image viewport with vertical scrolling, and full-source fallbacks should open near the saved evidence bbox rather than replacing the image with another crop.
- Overall-score visualization uses the five priority whole-star stages 3/4/5/6/7 with their backend-configured page names. The marker moves on the visible 3–7 scale, while half-star ratings remain available as the current floating value.
- Responsive score cards must preserve the first and last star labels inside the panel at phone widths and retain padded warning copy. Score values roll from zero to the active result, while the scale fill and marker animate to their calculated position; all motion must honor `prefers-reduced-motion` and replay when the selected diagnosis version changes.
- If diagnosis job creation or execution cannot reach the configured model or PDP Skill, show a blocking failure modal with the backend error and a route back to upload; never silently replace a forced production adapter with mock.
- Mock is test-only. Normal upload flows must block before job creation when `active_adapter=mock`; the backend must independently reject the request unless `PDP_ALLOW_MOCK_DIAGNOSIS=1` is explicitly set for automated tests.
- A configured API key is not proof of PDP readiness. Model verification must separately pass authentication, model discovery, text protocol, image/file ingestion, and structured 11-module output before the UI may call it usable for formal scoring.
- ModelVerse MiMo uses the OpenAI-compatible `chat_completions` protocol with image Data URLs. This path supports PNG/JPG but must reject PDF with a clear upload-format error; do not silently route PDFs through a non-working Files API.
- ModelVerse Claude-compatible model IDs beginning with `claude` reject the deprecated `temperature` parameter. Omit it, retain the validated JSON mode and supported thinking option, and use a 24000-token completion budget so complex long-page diagnoses can finish the structured payload.
- Moonshot Kimi K3 uses the exact model ID `kimi-k3` with the OpenAI-compatible `chat_completions` protocol. The production `api.moonshot.cn` gateway fixes temperature at 1, so omit the parameter; K3 always reasons, so never send the legacy `thinking.type=disabled` option. Keep its documented default 131072-token completion budget instead of forcing the legacy 12000-token cap, use top-level `reasoning_effort`, and use `json_object` plus Pydantic validation for the full PDP schema because Moonshot's stricter flavored JSON Schema rejects Pydantic's numeric coefficient enum.
- Keep Pydantic plus the server scoring engine as the hard safety boundary for every provider. Do not coerce malformed model fields or derive scores heuristically; a provider is ready only after all 11 modules, evidence coverage, and discrete coefficients validate.
- Persist evidence classifier names in a field wide enough for every supported enum value; the current contract is `PageEvidence.evidence_type` at 64 characters, including `pagewide_designed_model_product_sequence`.
- Preserve the legacy `/admin/diagnosis/integrationsettings/` redirect while SimpleUI users may still have the retired combined-settings tab cached; it must resolve to the independent AI model settings list instead of returning 404.
- Brand asset matching and AI image/page generation are deferred. Keep their information architecture visible only as clearly labeled gray disabled controls; do not present mock actions as usable capabilities.
- Login, registration, session recovery, profile editing, notification preferences, project creation, and PDP upload are baseline functions and must remain operational while the evaluation loop evolves.
- Public registration must always create an active ordinary user with both `is_staff=False` and `is_superuser=False`; ignore any client-supplied privilege fields. Only active superusers may authenticate to Django Admin. The frontend must show the admin entry only when `is_superuser=true`, while the backend admin permission and login form remain the hard authorization boundary.
- User identity presentation must come from the authenticated session on every surface. Never hardcode avatar initials, nickname, email, or role. Persist uploaded avatars in `UserProfile`, reuse the shared avatar component, and derive fallback initials from the current nickname/username when no image exists.
- Admin links in the frontend must use the same-origin `/admin/` path so local and production deployments resolve to their own backend; never hardcode a localhost admin URL into a production build.
- A newly registered account starts with an empty project list. Every project query and project-scoped upload/diagnosis action must be limited to `owner=request.user`; legacy ownerless records and other users' projects must never appear as shared starter content.
- Production must keep development defaults isolated behind `PDP_ENV`: require a stable strong Django secret, PostgreSQL, Redis, HTTPS host/origin settings, Gunicorn, Celery Worker, Nginx, persistent media storage, readiness checks, and CSRF protection. Never commit `.env.production`, API keys, SSH material, databases, or uploaded media.
- GitHub/ECS releases must be commit-addressed and health-checked before updating `/srv/pdp-lab/current`; retain rollback releases and treat irreversible database migrations as requiring an external snapshot.
- PDP Skill scoring consistency is versioned: `pdp-v2` remains the three-level rollback baseline and preserves `pdp-v1` for historical diagnoses. Never mutate either standard in place.
- PDP Skill `pdp-v6` is the active five-level standard and preserves `pdp-v1` through `pdp-v5` for historical diagnoses. It is pinned to `PDP Scoring Spec v4.2.0 (2026-07-24)` and adds deterministic boundary locks: campaign/editorial scenario imagery without explicit scene-benefit proof is capped at 0.75; fit at 0.75 requires a size chart plus at least two supporting dimensions, while 1 also requires personalized/body-type guidance, fit visualization or consumer evidence, and complete version comparison; generic platform service alone is 0, and limited product-specific service is capped at 0.25 or 0.5. Current observed score and improvement potential remain separate. Generic/empty evidence stays weak; one-dimensional evidence is capped at relatively weak; T2 is capped at medium; T1 is strong; extreme strong requires complete credible information plus matched T0 execution. Never rewrite locked historical versions during rule activation or rollback.
- Future rapid Skill imports must follow a staged contract: validate complete schema and source revision, show a module/rule diff and regression result, create an inactive version, then let a superuser activate it atomically with a rollback target. Do not overwrite a historical `ScoringStandard` in place.
