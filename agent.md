## 历史版本：3.19（PDP v3 五级成熟度评分版）

3.19 将内置评分标准升级为独立的 `pdp-v3`，不覆盖 `pdp-v1`、`pdp-v2` 或任何历史锁定诊断：

- 模块成熟度升级为 `弱 / 较弱 / 中 / 较强 / 强`，系数为 `0 / 0.25 / 0.5 / 0.75 / 1`；整体星级区间和 11 模块权重暂时保持不变。
- 模型结构新增信息完整度、T2/T1/T0视觉层级与图文结合度；后端确定性限制单一证据最高0.25、T2最高0.5、T1最高0.75，只有完整可信信息与T0匹配才能计1。
- 产品相关KV文案或棚拍/模特图从旧版直接归零改为最高0.25；通用标题、占位、模板、空壳、单独Logo仍为0。
- 数据库总分、模块系数和模块得分升级为两位小数；前端总览、证据页、人工修订和状态颜色均支持五档。
- 本地历史案例保守影子对比见 `docs/PDP_V3_FIVE_LEVEL_COMPARISON.md`；旧“强”在没有T0结构化证据时按T1重分类为“较强”，仅用于显示规则影响，不写回历史记录。

验证状态：Django diagnosis 34/34 测试、system check 与 Vite 生产构建通过；本地迁移 `0016_pdp_v3_five_level_scoring` 已应用，`pdp-v3` 为当前内置激活标准。桌面端总览、诊断详情和人工复核的标题/正文共用一致内容轴；手机端人工复核已修复为真正单列，左右卡片均使用相同居中页边距且无横向溢出。恢复时停用 `pdp-v3` 并重新启用 `pdp-v2`，不要删除评分标准或历史诊断。

## 历史版本：3.16（生产前端缓存与注册身份修复版）

3.16 修复生产浏览器在发布后继续使用旧 SPA 包、从而把新注册用户错误显示为 `PW` 且暴露旧版“管理后台”菜单的问题：

- 构建后的 JS 继续使用内容哈希，资产本身可安全缓存；只让作为资产清单的 HTML 入口参与重新验证，不牺牲资源加载性能。
- 注册测试新增断言：注册响应与随后 `/api/auth/me/` 的 `is_staff`、`is_superuser` 必须均为 `false`，覆盖数据库写入、会话和前端首屏所依赖的两条身份数据链路。

验证状态：Django 26/26 测试通过，Vite 生产构建通过。发布后应核查 `GET /index.html` 的 `Cache-Control: no-cache` 响应头、重新注册用户的头像首字母，以及普通用户菜单中不出现“管理后台”。

## 1. 项目目标

PDP Lab 是一个轻量 PDP 诊断、评分与优化工作台。使用者上传商品详情页后，可以快速理解评分内容、问题依据和后续优化方向，并继续完成：

1. PDP 内容上传与解析。
2. 11 个模块评分及整体星级判断。
3. P0 / P1 / P2 优化路线生成。
4. 品牌资产库匹配。
5. AI 优化版本生成。
6. 优化前后页面对比。
7. 复评分与下一轮优化。

未来正式版本需要能够直接接入品牌 DAM/资产库，并将已选品牌资产传给图像或页面生成服务。

## 7. 前后端适配约定

建议正式实现时保持以下对象边界：

- `Project`：项目名称、品牌、品类、最近版本、更新时间。
- `PdpSource`：图片、PDF、URL、上传状态、解析状态。
- `Diagnosis`：总分、星级、成熟度、11 模块结果、证据。
- `OptimizationTask`：优先级、模块、预计增益、状态、依赖资产。
- `BrandAsset`：DAM ID、标签、版权、匹配度、缩略图、原文件。
- `GenerationJob`：输入任务、资产 ID、Prompt、状态、多个版本。
- `OptimizedVersion`：生成结果、变更说明、复评分、版本关系。

品牌资产库接入时，前端不要直接依赖供应商返回结构；由后端建立统一的 `BrandAsset` 适配层。

## 8. 修改边界

- 不要因为恢复首页而删除工作台 Flow。
- 不要把整体星级用于单模块评分。
- 不要让首页重新出现工作台侧边栏。
- 不要删除“返回首页”和“快速切换项目”。
- 不要用静态占位替换已经存在的真实项目图片。
- 不要声明浏览器 QA 通过，除非实际打开页面、检查控制台并完成主流程交互。
- 修改 Figma 字体时，必须先确认字体可用并加载对应字重。

- 视觉与字体小修只更新 2.0 的“当前验证状态”。

- `agent.md` 是版本与恢复手册；`AGENTS.md` 只保存短小、长期有效的项目规则。
- 1.1 是当前本地交互版本；不要误将 2.0 的字体与首页样式重新覆盖到本地，除非用户明确要求“恢复 2.0”。
